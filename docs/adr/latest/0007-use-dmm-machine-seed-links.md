# 0007 - DMMぱちタウン由来の機種 seed と詳細 URL を使う

## Status

Proposed

## Context

現在の機種一覧は、`src/data/machines.ts` の静的 seed データを Dexie の `machines` テーブルへ同期する構成になっている。MVP では機種数が少なくても成立するが、実戦ログ入力や機種検索の使いやすさを上げるには、より広い機種一覧が必要になる。

DMMぱちタウンにはパチンコ/パチスロの機種一覧ページがあり、パチンコ一覧では機種名、メーカー、導入開始日、大当り確率などの概要と各詳細ページへのリンクが提供されている。

一方で、DMMぱちタウン利用規約では、サービス内容の変更や停止があり得ること、掲載情報の完全性・正確性・有用性などは保証されないこと、パチンコ機種/パチスロ機種情報を含むコンテンツの権利が DMM または許諾元に帰属することが示されている。

そのため、機種スペック、解析本文、画像などをアプリ内へ複製して保持するのではなく、ログ入力に必要な最小限の機種情報だけを seed として持ち、詳細は DMM の各ページ URL へ誘導する方式を検討する。

## Options Considered

### 1. 現在通り手書き seed を維持する

- **メリット**: 実装が最小で、外部サイトのページ構造や可用性に依存しない。
- **デメリット**: 機種追加が開発者の手作業になり、一覧の網羅性が低くなりやすい。

### 2. アプリ実行時に DMM から機種一覧を直接取得する

- **メリット**: ユーザーが最新に近い機種一覧を見られる可能性がある。
- **デメリット**: CORS、外部サイトの可用性、ページ構造変更、アクセス負荷、利用条件の影響をユーザー端末の通常利用に持ち込む。取得失敗時にログ入力体験へ影響しやすい。

### 3. DMM 一覧を手動スクリプトで取得し、最小 seed としてレビュー後に取り込む

- **メリット**: アプリ実行時は従来通り静的 seed と端末内 DB を使える。取得結果を差分レビューできる。DMM の詳細ページ URL へ誘導することで、詳細情報の複製を避けやすい。
- **デメリット**: 更新は手動になる。DMM 側のページ構造や URL が変わった場合、取得スクリプトや seed の更新が必要になる。

## Decision

DMM 由来の機種一覧は、アプリ実行時ではなく、開発時に手動スクリプトで取得して seed データへ反映する。

当面の取得対象はパチンコのみとする。ただし、既存の `MachineCategory` は `pachinko | slot | other` を持っているため、取得スクリプトとデータ構造は将来 `slot` を追加しやすい形にする。

seed に保持する項目は、まず `id`、`name`、`maker`、`category`、`dmmUrl` の最小構成とする。大当り確率、導入開始日、解析本文、画像などはアプリ内に保存しない。

更新は、必要なときに手動でスクリプトを実行し、生成差分をレビューして取り込む。CI などによる定期自動更新は当面行わない。

DMM 取得に失敗した場合は既存 seed を維持し、アプリのビルドや起動を止めない。更新スクリプトはエラーを報告するが、失敗時に seed ファイルを中途半端に変更しない。

`dmmUrl` は `/machines` の一覧または詳細プレビューに「DMMで詳細を見る」として表示する。ログ入力画面の機種選択では、入力速度を優先して DMM URL は表示しない。

## Consequences

機種一覧を広げやすくなり、ユーザーは必要に応じて DMM の詳細ページでスペックや解析情報を確認できる。アプリ本体はローカルファースト SPA と静的 seed 同期の構成を維持できる。

外部サイトの情報をアプリ内へ過度に複製しないため、更新負荷と権利面のリスクを抑えやすい。一方で、DMM 側のページ構造変更、URL 変更、サービス内容変更には追従が必要になる。

実装時には `Machine` 型、Dexie schema、seed 同期、機種画面、関連テストに `dmmUrl` を追加する必要がある。既存ログは `machineId` 参照のまま維持し、ログデータ自体へ DMM URL をコピーしない。

- DMMぱちタウン 利用規約: https://p-town.dmm.com/terms
