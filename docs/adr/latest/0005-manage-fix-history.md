# 0005 - 不具合と修正履歴を fix-log で管理する

## Status

Proposed

## Context

このアプリでは、MVP 実装後も GitHub Pages 公開、CI/CD、端末内保存、将来の AWS 移行など、運用中に再発しうる問題が出る。すでに `master` push で CI/CD が動かなかった件や、GitHub Pages 未有効化による deploy 失敗が発生している。

ADR は設計・運用方針の意思決定を残す場所であり、個々の不具合修正をすべて ADR 化すると重くなりすぎる。一方で、`backlog.md` は実装タスク管理、`docs/test-plan.md` は確認手順が主目的なので、「何が起きて、どう直して、どう確認したか」を追う専用の軽い履歴が必要になる。

## Options Considered

### 1. `docs/fix-log.md` で管理する

- **メリット**: ADR より軽く、Markdown で追いやすい。既存の `docs/test-plan.md` と同じ文脈に置ける。
- **デメリット**: 手動で書く必要があり、書き忘れの可能性がある。

### 2. ADR にすべて記録する

- **メリット**: 意思決定履歴と一体で残せる。
- **デメリット**: 小さな不具合修正まで ADR 化すると重すぎる。

### 3. GitHub Issues / PR だけで管理する

- **メリット**: 実際の開発フローと紐づけやすい。
- **デメリット**: 個人 MVP では運用が重くなりやすく、ローカルドキュメントだけ見たときに履歴が追いにくい。

## Decision

不具合・修正履歴は `docs/fix-log.md` で管理する。

記録対象は以下に限定する。

- ユーザー影響がある不具合
- CI/CD 失敗
- データ消失リスク
- 公開設定ミス
- 再発しそうな調査事項

小さな typo や軽微な見た目調整は記録しない。

1 件ごとの記録フォーマットは以下とする。

```markdown
## YYYY-MM-DD - タイトル

- 種別: ...
- 症状: ...
- 原因: ...
- 対応: ...
- 確認: ...
- 関連: ...
```

初期履歴として、以下の 2 件を `docs/fix-log.md` に記録する。

1. `master` push で CI/CD が動かなかった件
2. GitHub Pages 未有効化による `Get Pages site failed` 件

## Consequences

運用中の不具合や設定ミスについて、原因・対応・確認方法を軽く追えるようになる。再発時に過去の対応を参照しやすい。

一方で、手動記録なので書き忘れは起こりうる。すべての変更を網羅するものではなく、「再発時に役立つ履歴」として運用する。
