# 0003 - アプリの初期公開方法として GitHub Pages を採用する

## Status

Proposed

## Context

このアプリは `Vite + React + TypeScript` のローカルファースト SPA で、実戦ログは端末内 IndexedDB に保存する。MVP ではログインやクラウド同期を持たず、個人用に素早く使えることを優先する。

公開方法は、まず低コスト・低運用負荷で始めたい。一方で、将来的にはログイン、バックアップ、複数端末同期のために AWS 移行も視野に入れる。

GitHub Pages で公開する場合、第三者が URL を開くとアプリ自体は表示できる。ただし、MVP の個人データは公開サーバー上ではなく各端末の IndexedDB に保存されるため、URL アクセスだけで自分の実戦ログが直接見られるわけではない。

## Options Considered

### 1. GitHub Pages

- **メリット**: 無料で始めやすい。GitHub Actions で `npm test` / `npm run build` / デプロイを自動化できる。静的 SPA と相性がよい。
- **デメリット**: 基本的にサイト自体は公開され、指定ユーザーだけに限定する用途には弱い。アプリの存在・画面・フロントエンドコードは第三者に見られる可能性がある。

### 2. Cloudflare Pages / Vercel / Netlify + アクセス制限

- **メリット**: 静的 SPA のまま、メール認証などで入口を閉じやすい。
- **デメリット**: MVP では GitHub Pages より少し構成要素が増える。

### 3. AWS で初期公開

- **メリット**: 将来の `S3 + CloudFront`、`Cognito`、`API Gateway/Lambda/DynamoDB` などに自然につながる。
- **デメリット**: MVP 初期公開としては設定・運用が重い。

## Decision

MVP の初期公開は GitHub Pages を採用する。カスタムドメインは使わず、まずは `https://<user>.github.io/<repo>/` で公開する。

`main` ブランチにマージされたら GitHub Actions でテスト・ビルド・デプロイを自動実行する。公開対象は Vite のビルド成果物である `dist/` とする。

アクセス制限は初期対応しない。第三者が URL を開くとアプリ自体は表示できるが、実戦ログなどの個人データは各端末の IndexedDB に保存されるため、公開サーバー上には置かない。

将来的にログイン、バックアップ、複数端末同期が必要になったら、AWS への移行を検討する。候補は `S3 + CloudFront` による静的配信、`Cognito` による認証、`API Gateway/Lambda/DynamoDB` などによるクラウド保存・同期とする。

## Consequences

初期公開の手間と運用コストを小さくできる。GitHub Actions によって、テストとビルドを通った成果物だけを公開しやすい。

一方で、GitHub Pages 単体では指定ユーザーだけにアクセスを限定しにくい。アプリの存在・画面・フロントエンドコードは第三者に見られる可能性がある。ただし、MVP では個人データを端末内保存に限定するため、URL アクセスだけで自分の収支ログが漏れるリスクは低い。

将来 AWS へ移る可能性を考慮し、フロントエンドは静的配信可能な SPA として維持し、データアクセスは repository 層に閉じ込めておく。
