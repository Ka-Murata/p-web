# MVP 手動確認シナリオ

## 前提

- `npm install` 済み。
- `npm run dev` で http://localhost:5173/ を開く。
- MVP は端末内 IndexedDB に保存するため、ブラウザのサイトデータを消すとログも消える。

## 初回起動からログ追加

1. `/logs` を開き、ログなし状態と今月サマリーのゼロ表示を確認する。
2. 下部タブで `/machines` を開き、seed 機種が表示されることを確認する。
3. `/logs/new` を開き、日付、ホール名、機種、投資、回収を入力する。
4. 投資・回収を変えると収支プレビューが即時更新されることを確認する。
5. 保存し、`/logs` に戻って機種名、日付、ホール名、投資、回収、収支が表示されることを確認する。

## 編集・削除

1. `/logs` の実戦カードから「編集」を押す。
2. 既存値がフォームに入っていることを確認する。
3. 投資または回収を変更して保存し、一覧の収支が更新されることを確認する。
4. もう一度編集画面を開き、「削除」を押す。
5. 確認ダイアログで削除し、一覧から対象ログが消えることを確認する。

## 分析画面

1. 複数月、複数ホール、複数機種のログを追加する。
2. `/analytics` を開き、KPI、月別推移グラフ、ホール別ランキング、機種別ランキングが表示されることを確認する。
3. ログがない状態では分析データなし状態が表示され、画面が崩れないことを確認する。

## 補正済み平均回転数

1. `/logs/new` を開き、台番号/台識別、投資、貸し玉単価、通常回転へ使った出玉、回転数メモを入力する。
2. 入力中に平均回転プレビューが `回/1k` で即時更新され、補正投資額も表示されることを確認する。
3. 貸し玉単価と通常回転へ使った出玉を空欄にしても、補正なしの平均回転または安全な未計算表示になり、`NaN` や `Infinity` が表示されないことを確認する。
4. 保存後、編集画面で台番号/台識別、貸し玉単価、通常回転へ使った出玉、回転数メモが復元されることを確認する。
5. `/analytics` を開き、平均回転、補正投資、台別回転カードが表示されることを確認する。
6. 台番号/台識別があるログは同じ台単位、未入力ログはホール + 機種単位で集計されることを確認する。

## 機種画面

1. `/machines` を開き、機種名、メーカー、カテゴリ、メモを確認する。
2. 機種名またはメーカー名で検索し、一覧が絞り込まれることを確認する。
3. 検索入力を消すと一覧が戻ることを確認する。
4. 機種を選択し、詳細プレビューが更新されることを確認する。
5. DMM URL がある機種では「DMMで詳細を見る」が表示され、新しいタブで DMM 詳細ページを開けることを確認する。
6. DMM URL がない機種では空の詳細リンクが表示されないことを確認する。
7. `/logs/new` と編集画面の機種選択には DMM URL が表示されず、機種名だけで選べることを確認する。

## DMM seed 更新

1. 必要なときだけ `npm run seed:dmm -- --out tmp/dmm-machines.seed.ts` を手動実行する。
2. 取得に失敗した場合、既存の `src/data/machines.ts` が書き換わっていないことを確認する。
3. 生成された `tmp/dmm-machines.seed.ts` の差分をレビューし、`id`、`name`、`maker`、`category`、`dmmUrl` 以外の項目が含まれていないことを確認する。
4. アプリ内に大当り確率、導入開始日、解析本文、画像を保存しないことを確認する。
5. 取り込む機種だけを `src/data/machines.ts` に反映し、`npm test` と `npm run build` を実行する。

## モバイル表示

1. ブラウザ DevTools で幅 390px 前後にする。
2. `/logs`、`/logs/new`、`/analytics`、`/machines` を確認する。
3. 長いホール名、長い機種名、マイナス収支でも文字やボタンが重ならないことを確認する。
4. 下部タブと保存ボタンがタップしやすく、フォーム入力を隠し続けないことを確認する。

## 黒ベーステーマ

1. `/logs` を開き、背景、今月サマリー、端末内保存案内、実戦カード、下部タブが黒・濃いグレー基調で表示されることを確認する。
2. `/logs/new` と `/logs/:id/edit` を開き、入力欄、収支プレビュー、回転数メモ、保存ボタン、削除ダイアログが黒背景で読めることを確認する。
3. プラス収支、マイナス収支、ゼロ収支を用意し、勝敗色が黒背景で判別できることを確認する。
4. `/analytics` を開き、KPI、月別推移グラフ、ランキング、データなし状態の文字・軸・棒色が読めることを確認する。
5. `/machines` を開き、検索欄、選択中プレビュー、DMM 詳細リンク、機種一覧の選択状態が黒背景で読めることを確認する。
6. ロゴ画像が従来どおりヘッダーに表示され、黒ベース化で差し替わっていないことを確認する。

## GitHub Pages 公開後確認

1. GitHub のリポジトリ設定で Pages の source が GitHub Actions になっていることを確認する。
2. `master` ブランチへの push 後、GitHub Actions の `Pages CI/CD` workflow で `quality` と `deploy` が成功していることを確認する。
3. `quality` job で `npm ci`、`npm run check:public-safety`、`npm audit --audit-level=high`、`npm test`、`npm run build` が成功していることを確認する。
4. workflow の結果に表示された公開 URL を開き、アプリが表示されることを確認する。
5. 公開 URL の `/logs`、`/analytics`、`/machines` をタブ操作で開けることを確認する。
6. `/logs` で `端末内保存` バッジと端末内保存の案内が表示されることを確認する。
7. 実戦ログを追加し、同じ端末・同じブラウザでリロードしてもデータが残ることを確認する。
8. 別ブラウザまたは別端末で公開 URL を開き、自分の実戦ログが表示されないことを確認する。
9. GitHub Pages 単体では指定ユーザー制限をしないため、URL を知っている第三者はアプリ自体を開けることを前提として確認する。

## GitHub Pages CI/CD 失敗確認

1. `.env.example` はコミット対象に含めても `npm run check:public-safety` が成功することを確認する。
2. `.env`、秘密鍵ファイル、`AWS_SECRET_ACCESS_KEY=`、`GITHUB_TOKEN=`、`PASSWORD=`、`TOKEN=` などを含めると `npm run check:public-safety` が失敗することをローカルで確認する。確認後は該当ファイルや文字列を残さない。
3. `npm audit --audit-level=high` が high/critical の脆弱性を検出した場合、`quality` job が失敗し、`deploy` が実行されないことを確認する。
4. `npm test` または `npm run build` が失敗した場合、`deploy` が実行されないことを確認する。

## 既知の未対応事項

- クラウド同期、ログイン、CSV 入出力は MVP 後。
- 機種 seed は DMM 詳細 URL までの最小データで、詳細スペック管理は未対応。
- E2E テストは未導入。MVP では Vitest と手動確認で担保する。
- ホール名の表記ゆれ統合は未対応。
