# AIを用いた個人開発で「勝手に進ませない」ための開発フロー

## 発表の狙い

このプロジェクトを通して、AIを使った個人開発では「速く作る」だけでなく、「決めたことから実装がズレない仕組み」が重要だと感じた。

特に、ADR、画面レイアウト、backlog、実装、修正履歴をどう接続するかが課題だった。

## 背景

個人開発でAIを使うと、実装速度はかなり上がる。

一方で、次のような問題が起きた。

- AIがその場でよさそうなUIや仕様を足してしまう
- `layout.pen` で決めた画面と実装が少しずつズレる
- 修正や不具合対応が、設計フローの外側で進んでしまう
- 会話ログ上では納得していても、後から再開すると現在地が分からない

最初は `grill-to-adr` と `adr-to-layout-backlog` がかなり良かった。

- 何を作るかを問答で決める
- 決定を ADR に残す
- ADR を `layout.pen` に反映する
- 承認後に backlog 化する

ただし、その後の実装フェーズで `layout.pen` と実装が乖離していく問題が残った。

## 問題意識

本来は `layout.pen` が画面仕様の正であるべき。

しかし実装に入ると、AIはコードの文脈だけを見て判断しがちになる。

その結果、以下が起きる。

```txt
ADRでは決めた
layout.penにも反映した
backlogにもした
でも実装は少し違う
```

これは嫌だった。

なぜなら、`layout.pen` が「参考資料」になってしまうと、設計した意味が薄れるから。

## 方針

解決方針は、`layout.pen` を「参考」ではなく「実装契約」にすること。

そのために、開発フローを以下のように分けた。

```txt
新機能・設計判断:
grill-to-adr
  → adr-to-layout-backlog
  → layout-to-implementation
  → design-drift-audit

不具合・修正要求:
fix-log-repair-flow
  → fix-to-layout-backlog
  → layout-to-implementation
  → design-drift-audit
```

ポイントは、不具合修正も追加機能と同じ流れに乗せること。

UIに影響する修正なら、いきなりコードを直さない。

まず「本来どうあるべきか」を決め、`layout.pen` に反映し、承認し、backlog 化してから実装する。

## 作ったスキル

### grill-to-adr

新機能・設計判断の入口。

一問ずつ深掘りして、何を作るか、なぜ作るか、何を後回しにするかを ADR に残す。

### adr-to-layout-backlog

ADR を `layout.pen` に反映し、承認後に backlog へ落とす。

backlog には `Design Contract` を持たせる。

```md
- Design Contract:
  - ADR: docs/adr/latest/NNNN-xxx.md
  - Layout: docs/screen/layout.pen#screen-or-state
  - 本来の状態: ...
  - Out of Scope: ...
```

### fix-log-repair-flow

不具合・修正要求の入口。

`Repair Grill` を追加し、修正方針を問答で決める。

聞くことは軽量にした。

- 何が壊れている状態か
- 本来どうなっていれば正しいか
- 影響範囲はどこまでか
- 修正案は何か
- 今回どこまで直すか

### fix-to-layout-backlog

UI影響ありの不具合を、`layout.pen` 更新、承認、backlog 化へ昇華する。

修正も「小さな変更要求」として扱う。

### layout-to-implementation

承認済み `layout.pen` と backlog の `Design Contract` を正として実装する。

実装中に layout と違う判断が必要になったら、実装で勝手に決めず、上流に戻る。

### design-drift-audit

`layout.pen` と実装UIの乖離を検査する。

乖離があれば分類する。

- 実装漏れ
- 実装過剰
- 情報設計ズレ
- 状態不足
- 文言差分
- layout陳腐化
- ADR違反

そして戻し先を決める。

## 上位オーケストレーション

個別スキルを毎回呼ぶのは面倒なので、上位スキルも作った。

```txt
feature-flow
repair-flow
```

ただし、上位スキルは個別スキルを置き換えない。

役割は以下だけ。

- 現在地の確認
- 次に使うスキルの判断
- 成果物の受け渡し
- 承認ゲートで止まること
- 途中再開時のルーティング

つまり、呼び出しは便利にするが、判断の責務は混ぜない。

## 分けることの意味

一見すると、一つの巨大スキルで全部やれば便利に見える。

でも、あえて分ける強みがあった。

- 判断の種類を混ぜない
- 承認ゲートを置ける
- 途中成果物から再開できる
- AIの勝手な仕様追加を抑えられる
- 実装者と監査者の役割を分けられる

これは、同じAIに複数の帽子をかぶらせるようなもの。

設計者、画面設計者、実装者、レビュアーを分けることで、AIの出力が混線しにくくなる。

## 得られた考え方

AI個人開発では、AIに全てを一気通貫で任せるより、成果物ごとに状態を持たせる方が扱いやすい。

```txt
ADR: 何を決めたか
layout.pen: 画面としてどうあるべきか
backlog: 何を実装するか
実装: どう作ったか
fix-log: 何が壊れ、どう直したか
drift audit: 決めたことからズレていないか
```

この状態が残っていると、翌日でも、別セッションでも、別AIでも再開しやすい。

## まとめ

AIを使うと、コードを書く速度は上がる。

しかし、速く作れるからこそ、どこで止まるか、何を正とするか、どの成果物を次の入力にするかが重要になる。

今回の結論はこれ。

```txt
AIに速く進ませるためではなく、
AIが勝手に進まないためにフローを作る。
```

`layout.pen` を正にし、ADR、fix-log、backlog、実装、監査を接続することで、個人開発でも設計判断を保ったままAIと進めやすくなった。

## 話すときの一言メモ

- 最初は「AIで速く作る」ことに意識が向いていた
- 途中から「AIが勝手に作らないようにする」ことが重要になった
- `layout.pen` が実装からズレるのが一番嫌だった
- 不具合修正も設計フローの外に出すと、そこから崩れる
- だから修正も layout 承認と backlog 化に乗せた
- 上位スキルは便利さ、個別スキルは責務分離
- AI個人開発に必要なのは、プロンプトだけでなく状態管理
