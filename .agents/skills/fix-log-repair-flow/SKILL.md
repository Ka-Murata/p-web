# fix-log-repair-flow

`docs/fix-log.md` を使って、不具合や運用トラブルを「何が起きたか、本来どうあるべきか、どう直すか、どう確認したか」まで追える形で扱う。UI に見える修正は、追加機能と同じく layout 承認と backlog 化へ進める。

## 使うタイミング

- ユーザー影響がある不具合を調査・修正するとき。
- 修正要求を「小さな変更要求」として扱いたいとき。
- CI/CD 失敗、GitHub Pages 公開失敗、設定ミスを直すとき。
- データ消失リスクや再発しそうな調査事項を扱うとき。
- ユーザーが「fix-log に記録して」「修正履歴を残して」「原因も残して」と言ったとき。

小さな typo や軽微な見た目調整だけなら、このスキルは必須ではない。

## ワークフロー

8 フェーズで進める。

### Phase 1: Triage（切り分け）

1. ユーザーの報告、エラーログ、再現条件を読む。
2. 関連ファイル、最近の変更、既存の `docs/fix-log.md` を確認する。
3. 記録対象か判定する。対象は以下:
   - ユーザー影響がある不具合
   - UI 影響がある修正要求
   - CI/CD 失敗
   - データ消失リスク
   - 公開設定ミス
   - 再発しそうな調査事項
4. UI 影響の有無を分類する。
   - `UI影響あり`: layout 更新・承認・backlog 化へ進める。
   - `UI影響なし`: fix-log から直接修正できる。
5. 不明点が修正方針に影響する場合だけ、ユーザーに短く確認する。コードやログから分かることは自分で調べる。

### Phase 2: Draft Log（仮記録）

修正に入る前に、`docs/fix-log.md` に記録する骨子を作る。すぐに原因が確定しない場合は、調査メモとして保持し、確定時に反映する。

```markdown
## YYYY-MM-DD - タイトル

- 種別: 不具合 / 修正要求 / Layout乖離 / ADR違反 / CI/CD / 公開設定ミス / データ / 調査
- 状態: 調査中
- UI影響: あり / なし
- 症状: ...
- 本来の状態: 未確認
- 原因: 調査中
- 修正方針: 未決定
- 対応Layout: 未作成
- 対応Backlog: 未作成
- 対応: 未対応
- 確認: 未確認
- 関連: ...
```

既存の同じ問題が `docs/fix-log.md` にある場合は、新規項目ではなく既存項目に追記する。

### Phase 3: Repair Grill（修正方針の深掘り）

`grill-to-adr` と同じく、一問ずつ修正方針を決める。最大 5 問を目安にする。

**質問の観点**:

- 何が壊れている状態か？
- 本来どうなっていれば正しいか？
- 影響範囲はどこまでか？
- 修正案は何があるか？
- 今回どこまで直し、何を後回しにするか？

**ルール**:

- コードやログから分かることは自分で調べる。
- ユーザーには「本来の状態」や「修正範囲」の判断だけを聞く。
- 各質問に推奨回答を添える。
- ユーザーが「もういい」「十分」「まとめて」と言ったら Phase 4 に進む。

### Phase 4: Fix Decision（修正方針の確定）

Repair Grill の結果を fix-log に記録する。

```markdown
- 状態: 方針決定
- 本来の状態: ...
- 修正案:
  1. ...
  2. ...
- 決定: ...
- 理由: ...
- 今回直す範囲: ...
- 後回し: ...
```

ADR に昇格する条件:

- 既存仕様を変える。
- データモデルを変える。
- 画面全体の情報設計を変える。
- 今後の開発ルールを変える。
- 複数案のトレードオフが重い。

該当する場合は修正を止め、`grill-to-adr` を使う。

### Phase 5: Layout / Backlog Gate

`UI影響あり` の場合:

1. 直接実装せず、`fix-to-layout-backlog` に進む。
2. `layout.pen` に正しい画面状態を反映する。
3. ユーザー承認後、`backlog.md` に Design Contract 付きタスクを追加する。
4. fix-log の状態を `Backlog化済み` にする。

`UI影響なし` の場合:

- Phase 6 に進み、最小変更で修正する。

### Phase 6: Repair（修正）

1. UI 影響ありの場合は、承認済み backlog を対象に `layout-to-implementation` を使う。
2. UI 影響なしの場合は、最小の変更で原因を直す。
3. 関連するテスト、設定、ドキュメントも必要な範囲だけ更新する。
4. ユーザーの未コミット変更を戻さない。
5. 既存 ADR の決定を変える必要が出た場合は、修正を止めて `grill-to-adr` を提案する。

### Phase 7: Verify（検証）

修正内容に応じて確認する。

- React/UI: `npm test`、必要なら `npm run build`、必要なら `design-drift-audit`。
- CI/CD・公開設定: `npm run check:public-safety`、`npm audit --audit-level=high`、`npm run build`、workflow 条件確認。
- DB/保存: repository/domain テストと、データ移行・既存データ読み込みの確認。
- ドキュメントのみ: 関連リンク、日付、ファイル名、手順の整合性確認。

検証できなかった項目は、理由を `確認` に明記する。

### Phase 8: Record（確定記録）

`docs/fix-log.md` に、確定した内容を追記または更新する。最新の問題を上に置く。

```markdown
## YYYY-MM-DD - タイトル

- 種別: CI/CD / 公開設定ミス / 不具合 / 修正要求 / Layout乖離 / ADR違反 / データ / 調査
- 状態: 修正済み / Backlog化済み / 保留
- UI影響: あり / なし
- 症状: ユーザーやCIが観測した事実。
- 本来の状態: どうなっていれば正しいか。
- 原因: 確認できた直接原因。推測なら「可能性が高い」と書く。
- 修正方針: Repair Grill で決めた方針。
- 対応Layout: layout 参照。UI影響なしなら不要。
- 対応Backlog: backlog 参照。UI影響なしなら不要。
- 対応: 実際に変更した内容。
- 確認: 実行したコマンド、確認した画面、未確認事項。
- 再発防止: 追加したテスト、checklist、今後の注意点。
- 関連: 変更ファイル、ADR、backlog、issue/PR/commit があれば書く。
```

## ルール

- `docs/fix-log.md` は再発防止と修正要求管理の履歴であり、全変更の網羅ログではない。
- 原因が未確定なら断定しない。観測事実と推測を分ける。
- 修正していない問題を「対応済み」と書かない。
- 検証していないコマンドを実行済みのように書かない。
- UI に見える変更は、原則 `layout.pen` 更新・承認・backlog 化を通す。
- 軽微な typo、明らかな実装ミス、小さなスタイル崩れは直接修正してもよいが、理由を共有する。
- 同じ問題が再発したら、既存項目に「再発」メモを足すか、新しい発生日の項目から既存項目へ関連付ける。
- ユーザーが「記録だけ」と言った場合は、修正せず `docs/fix-log.md` だけ更新する。

## 完了条件

- 原因または暫定原因が説明されている。
- Repair Grill により修正方針が決まっている。
- UI 影響ありなら layout 承認と backlog 化に進んでいる、または未実施理由が明記されている。
- 必要な修正が実装されている、または未対応理由が明記されている。
- 適切な検証が実行され、結果がユーザーに共有されている。
- `docs/fix-log.md` に症状・本来の状態・原因・修正方針・対応・確認・関連が記録されている。

## 併用するスキル

- 方針変更や運用ルールの決定が必要なら `grill-to-adr` を使う。
- UI 影響ありの修正を layout/backlog へ落とすなら `fix-to-layout-backlog` を使う。
- 承認済み layout/backlog を実装するなら `layout-to-implementation` を使う。
- 実装後に layout と実装の乖離を確認するなら `design-drift-audit` を使う。
- React 実装を伴う修正なら `vercel-react-best-practices` を併用する。
