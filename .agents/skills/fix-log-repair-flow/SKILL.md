# fix-log-repair-flow

`docs/fix-log.md` を使って、不具合や運用トラブルを「何が起きたか、なぜ起きたか、どう直したか、どう確認したか」まで追える形で修正するための手順。

## 使うタイミング

- ユーザー影響がある不具合を調査・修正するとき。
- CI/CD 失敗、GitHub Pages 公開失敗、設定ミスを直すとき。
- データ消失リスクや再発しそうな調査事項を扱うとき。
- ユーザーが「fix-log に記録して」「修正履歴を残して」「原因も残して」と言ったとき。

小さな typo や軽微な見た目調整だけなら、このスキルは使わない。

## ワークフロー

5 フェーズで進める。

### Phase 1: Triage（切り分け）

1. ユーザーの報告、エラーログ、再現条件を読む。
2. 関連ファイル、最近の変更、既存の `docs/fix-log.md` を確認する。
3. 記録対象か判定する。対象は以下:
   - ユーザー影響がある不具合
   - CI/CD 失敗
   - データ消失リスク
   - 公開設定ミス
   - 再発しそうな調査事項
4. 不明点が修正方針に影響する場合だけ、ユーザーに短く確認する。コードやログから分かることは自分で調べる。

### Phase 2: Draft Log（仮記録）

修正に入る前に、`docs/fix-log.md` に記録する内容の骨子を作る。すぐに原因が確定しない場合は、最終追記まで待ってもよいが、調査メモとして以下を手元で保持する。

```markdown
## YYYY-MM-DD - タイトル

- 種別: ...
- 症状: ...
- 原因: 調査中
- 対応: 未対応
- 確認: 未確認
- 関連: ...
```

既存の同じ問題が `docs/fix-log.md` にある場合は、新規項目ではなく既存項目に追記する。

### Phase 3: Repair（修正）

1. 最小の変更で原因を直す。
2. 関連するテスト、設定、ドキュメントも必要な範囲だけ更新する。
3. ユーザーの未コミット変更を戻さない。
4. 既存ADRの決定を変える必要が出た場合は、修正を止めて `grill-to-adr` の利用を提案する。

### Phase 4: Verify（検証）

修正内容に応じて確認する。

- React/UI: `npm test`、必要なら `npm run build`。
- CI/CD・公開設定: `npm run check:public-safety`、`npm audit --audit-level=high`、`npm run build`、workflow 条件確認。
- DB/保存: repository/domain テストと、データ移行・既存データ読み込みの確認。
- ドキュメントのみ: 関連リンク、日付、ファイル名、手順の整合性確認。

検証できなかった項目は、理由を `確認` に明記する。

### Phase 5: Record（確定記録）

`docs/fix-log.md` に、確定した内容を以下の形式で追記または更新する。最新の問題を上に置く。

```markdown
## YYYY-MM-DD - タイトル

- 種別: CI/CD / 公開設定ミス / 不具合 / データ / 調査 など
- 症状: ユーザーやCIが観測した事実。
- 原因: 確認できた直接原因。推測なら「可能性が高い」と書く。
- 対応: 実際に変更した内容。
- 確認: 実行したコマンド、確認した画面、未確認事項。
- 関連: 変更ファイル、ADR、backlog、issue/PR/commit があれば書く。
```

## ルール

- `docs/fix-log.md` は再発防止のための履歴であり、全変更の網羅ログではない。
- 原因が未確定なら断定しない。観測事実と推測を分ける。
- 修正していない問題を「対応済み」と書かない。
- 検証していないコマンドを実行済みのように書かない。
- 同じ問題が再発したら、既存項目に「再発」メモを足すか、新しい発生日の項目から既存項目へ関連付ける。
- ユーザーが「記録だけ」と言った場合は、修正せず `docs/fix-log.md` だけ更新する。

## 完了条件

- 原因または暫定原因が説明されている。
- 必要な修正が実装されている、または未対応理由が明記されている。
- 適切な検証が実行され、結果がユーザーに共有されている。
- `docs/fix-log.md` に症状・原因・対応・確認・関連が記録されている。

## 併用するスキル

- 方針変更や運用ルールの決定が必要なら `grill-to-adr` を使う。
- ADR の反映をレイアウトやバックログへ落とすなら `adr-to-layout-backlog` を使う。
- React 実装を伴う修正なら `vercel-react-best-practices` を併用する。
