# grill-to-adr

grill-me スタイルの深掘りインタビューで意思決定を引き出し、合意に達したら ADR を生成する。新機能・設計判断の正式な入口として使う。

## ワークフロー

3フェーズで進行する。

### Phase 1: Grill（深掘り）

ユーザーが提示したテーマについて、以下の観点で一問ずつ深掘りする。

**質問の観点**:

- 背景・動機: なぜこの決定が必要か？
- 選択肢: 他にどんな選択肢を検討したか？
- 比較軸: 各選択肢のトレードオフは？
- 制約: 技術的・時間的・運用上の制約は？
- 影響範囲: 影響する画面、データ、コンポーネント、将来の決定は？
- MVP: 今回作る範囲と後回しにする範囲は？
- 撤回コスト: この決定を後から変えるコストはどの程度か？

**ルール**:

- **一問ずつ** 聞く。まとめて聞かない。
- 各質問に **推奨回答** を添える。
- コードベースを調べれば答えられる質問は、コードベースを調べて自分で答える。
- ユーザーが「もういい」「十分」「まとめて」と言ったら Phase 2 に移行する。
- 最大 10 問を目安とし、合意が見えたら切り上げを提案する。

### Phase 2: Summarize（要約・確認）

Grill で得た情報を ADR 構造に整理して提示する。

```markdown
タイトル案: {タイトル}（slug: {slug候補}）
Status: Proposed
Context: {背景・制約・動機}
Options Considered:
  1. {選択肢A} — メリット / デメリット
  2. {選択肢B} — メリット / デメリット
Decision: {選んだ選択肢とその理由}
Consequences: {良い影響 / 悪い影響 / 受け入れるトレードオフ}
Implementation Notes: {layout/backlog/実装で守る制約。あれば}
Out of Scope: {後回しにする範囲。あれば}
```

- 「この内容で ADR を作成してよいか？」とユーザーに確認する。
- 修正があれば反映して再提示する。

### Phase 3: Record（ADR 生成）

ユーザーの承認後、以下のルールに従って ADR を生成する。

**ディレクトリ構成**:

```text
docs/adr/
├── latest/
│   └── NNNN-{slug}.md
└── archive/
    └── NNNN/
        └── {slug}-YYYYMMDD.md
```

**手順**:

1. `docs/adr/latest/` 内の最大番号を確認し、+1 で発番する（なければ `0001`）。
2. `docs/adr/latest/NNNN-{slug}.md` を作成する。
3. Phase 2 の内容を各セクションに流し込む。
4. Status は `Proposed` で開始する（ユーザーが `Accepted` を希望すればそちらにする）。
5. 次の工程として `adr-to-layout-backlog` を案内する。

## 既存 ADR の更新が必要な場合

**編集を開始する前に必ずユーザーに更新か上書きかを確認する。**

| 変更内容 | 推奨 |
| --- | --- |
| typo・てにをは・改行調整 | 上書き |
| 言い回しの改善、説明の追記（決定は不変） | 上書き |
| 技術スタックの追加・削除・差し替え | 更新 |
| データモデルのフィールド追加・削除・型変更 | 更新 |
| 画面全体の情報設計変更 | 更新 |
| ステータス変更（Accepted → Superseded など） | 更新 |

- **更新**: 旧版を `docs/adr/archive/NNNN/{slug}-YYYYMMDD.md` に退避 → 新版を latest に書く。
- **上書き**: archive 送りなし。直接編集する。

## ステータス値

- **Proposed**: 提案中。まだ合意されていない。
- **Accepted**: 合意済み。現在有効な決定。
- **Deprecated**: 非推奨。新規適用は避けるが既存は残る。
- **Superseded**: 別の決定に置き換えられた。

## ADR テンプレート

```markdown
# NNNN - {タイトル}

## Status

Proposed

## Context

{この決定が必要になった背景・課題を記述する。}

## Options Considered

### 1. {選択肢A}

- **メリット**: ...
- **デメリット**: ...

### 2. {選択肢B}

- **メリット**: ...
- **デメリット**: ...

## Decision

{何を決定したか。選んだ選択肢とその理由を記述する。}

## Consequences

{この決定による影響・トレードオフを記述する。良い影響と悪い影響の両方を含める。}

## Implementation Notes

{layout/backlog/実装で守る制約があれば記述する。}

## Out of Scope

{今回やらないことがあれば記述する。}
```

## 下流工程

- ADR を layout と backlog に落とすときは `adr-to-layout-backlog` を使う。
- 承認済み backlog を実装するときは `layout-to-implementation` を使う。
- 実装後の乖離確認には `design-drift-audit` を使う。
